# Техническое задание

## Система подбора киберспортсменов: сегментация, кластеризация, подбор оптимального состава команды

Версия 1.0. Практическая часть курсовой работы. Источники данных: Dota 2 (OpenDota API), CS2 (FACEIT Data API).

---

## 1. Общие сведения

### 1.1 Назначение

Локально работающая система, которая автоматически собирает статистику профессиональных/рейтинговых игроков Dota 2 и CS2 из открытых API, приводит её к единой схеме, вычисляет внутренний рейтинг силы каждого игрока, кластеризует игроков по стилю/уровню игры (сегментация) и по запросу подбирает оптимальный состав команды из 5 человек под заданные ограничения.

### 1.2 Основание для разработки

Курсовая работа «Система подбора киберспортсменов, их сегментация, кластеризация, система выстраивания оптимальной команды для побед на различных чемпионатах».

### 1.3 Область применения

Учебный прототип. Не предназначен для промышленной эксплуатации, коммерческого скаутинга или обхода лимитов и правил использования API источников данных.

## 3. Цель и задачи разработки

**Цель:** показать на практике полный цикл — от сбора сырых данных об игроках до выдачи рекомендованного состава команды — как реализацию темы курсовой работы.

**Задачи:**

1. Автоматически собирать и обновлять данные об игроках Dota 2 и CS2 из открытых API без ручного ведения списков.
2. Хранить историю показателей игрока во времени (не только последний срез).
3. Сегментировать/кластеризовать игроков по стилю и уровню игры на основе накопленных метрик.
4. Рассчитывать прозрачный, объяснимый рейтинг силы игрока.
5. Подбирать оптимальный состав из 5 игроков под ограничения (роль/кластер) методом целочисленного программирования.
6. Предоставить локальный веб-интерфейс для просмотра игроков, кластеров, рейтингов и запроса оптимального состава.

## 4. Архитектура системы

### 4.1 Общая схема потока данных

```
[Планировщик, APScheduler]
        │  раз в N минут
        ▼
[Адаптеры источников]  OpenDota (Dota 2)  /  FACEIT Data API (CS2)
        │  сырые JSON-ответы
        ▼
[Нормализация]  →  единая схема игрока/снапшота
        │
        ▼
[Хранилище: SQLite]  players, player_snapshots
        │
        ├──────────────┬───────────────────┐
        ▼              ▼                   ▼
 [Рейтинг]      [Кластеризация]     [FastAPI: чтение сырых данных]
        │              │
        └──────┬───────┘
               ▼
      [Оптимизатор состава, PuLP]
               │
               ▼
      [Streamlit-дашборд]  ← читает всё через FastAPI или напрямую из БД
```

### 4.2 Модульность

Каждый модуль — независимый компонент с чётким интерфейсом входа/выхода (см. раздел 6). Коллектор данных не знает о существовании кластеризации; оптимизатор не знает, откуда взялся рейтинг. Это отдельное требование: добавление новой игры или замена алгоритма кластеризации не должны требовать изменений в других модулях.

## 5. Функциональные требования по модулям

### 5.1 Модуль сбора данных (`collectors/`)

- Реализовать адаптер для OpenDota API (Dota 2) и адаптер для FACEIT Data API (CS2) с единым интерфейсом: `fetch_player_pool()` — список кандидатов, `fetch_player_stats(external_id)` — статистика конкретного игрока.
- Формирование пула игроков — **автоматическое**, без ручного ввода:
  - Dota 2: список профессиональных игроков через эндпоинт OpenDota `proPlayers` (проверить актуальное название/URL в текущей документации docs.opendota.com — на момент подготовки ТЗ страница со Swagger-описанием не отдавала статический контент при автоматической проверке, финальный URL нужно подтвердить перед реализацией).
  - CS2: FACEIT Data API v4 (`https://open.faceit.com/data/v4`), подтверждённые эндпоинты: `GET /players/{player_id}`, `GET /players/{player_id}/games/{game_id}/stats`, `GET /players/{player_id}/history`, `GET /leaderboards/{leaderboard_id}`, `GET /leaderboards/{leaderboard_id}/players/{player_id}`, `GET /players` (поиск по nickname). Готового «топ-N глобально» эндпоинта без указания конкретного `leaderboard_id`/`hub_id` в API нет — конкретный `leaderboard_id` для формирования пула (например, официальный CS2 ranked leaderboard) нужно определить на этапе реализации и задать конфигурацией, а не хардкодить как факт в этом ТЗ.
- Соблюдать лимиты запросов каждого источника (обрабатывать HTTP 429, делать экспоненциальный backoff).
- При недоступности одного источника — не блокировать сбор по другому источнику (изоляция ошибок на уровне адаптера).
- Каждый успешный опрос игрока пишет новый снапшот, а не перезаписывает предыдущий.

### 5.2 Модуль нормализации (`normalize.py`)

- Преобразовать разноформатные ответы OpenDota и FACEIT в единую схему снапшота (см. раздел 6.2).
- Для Dota 2 роль (позиция 1–5) определять по доступным данным о линии/герое (например, поле роли/лейна в матчевой статистике OpenDota), агрегируя по последним матчам большинством голосов. Поле может быть `null`, если данных недостаточно — это допустимо.
- Для CS2 нативного поля «роль» в источнике нет: роль не собирается, а вычисляется позже кластеризацией (см. 5.3) — в схеме снапшота для CS2 роль изначально `null`.

### 5.3 Модуль сегментации/кластеризации (`clustering.py`)

- На вход — таблица признаков игрока, агрегированная из снапшотов за выбранный период (например, средние/медианные значения KDA, винрейта, урона, экономики и т.д., отдельно по каждой игре — Dota 2 и CS2 кластеризуются раздельно, признаки разные).
- Нормализация признаков (стандартизация/z-score) перед кластеризацией.
- Алгоритмы: KMeans как основной (число кластеров подбирается методом локтя/силуэта), DBSCAN — опционально, для сравнения и обнаружения выбросов.
- Для Dota 2 кластеры — уточнение/проверка формальных ролей 1–5. Для CS2 кластеры используются как псевдо-роли (playstyle-группы) в модуле оптимизации, так как формальных ролей источник не даёт.
- Опционально: PCA/t-SNE для визуализации кластеров в 2D на дашборде.
- Результат кластеризации сохраняется в таблицу `clusters` с привязкой к игроку, алгоритму и моменту пересчёта.

### 5.4 Модуль рейтинга (`rating.py`)

Простая, объяснимая модель (не ML): взвешенная сумма нормализованных показателей.

1. Для каждого показателя (например, для Dota 2: winrate, KDA, GPM, XPM, урон по героям; для CS2: winrate, K/D, ADR, HS%, число раундов с impact-действиями — набор фиксируется отдельно на этапе реализации по факту доступных полей API) вычисляется z-score относительно остальных игроков той же игры (и роли/кластера, если применимо).
2. Итоговый рейтинг — взвешенная сумма z-score с конфигурируемыми весами (задаются константой в коде, можно вынести в конфиг).
3. Рейтинг пересчитывается при каждом обновлении снапшотов и сохраняется в таблицу `ratings` с меткой времени — чтобы можно было показать историю изменения силы игрока на дашборде.

ML-модель прогнозирования вероятности победы состава по историческим матчам — вне текущей версии ТЗ, зафиксирована как возможное расширение в разделе 10.

### 5.5 Модуль оптимизации состава (`optimizer.py`)

Целочисленное программирование (PuLP), классическая задача выбора подмножества под ограничения — по аналогии с оптимизацией состава в fantasy-спорте.

**Переменные:** `x_i ∈ {0,1}` для каждого кандидата `i` из пула игроков одной игры (не смешивать Dota 2 и CS2 в одном составе).

**Целевая функция:** максимизировать `Σ rating_i * x_i`.

**Ограничения:**

- `Σ x_i = 5` — размер состава.
- Dota 2: по одному игроку на каждую из ролей 1–5 (`Σ x_i [role = r] = 1` для каждой роли `r`), если роль не определена у части кандидатов — их роль уточняется вручную перед запуском оптимизации или они исключаются из пула кандидатов на этот запуск.
- CS2: не более `MAX_PER_CLUSTER` игроков из одного кластера (параметр, по умолчанию 2) и представлено не менее `MIN_CLUSTERS` разных кластеров (параметр, по умолчанию 3) — так как жёстких ролей нет, ограничение обеспечивает разнообразие стилей игры в составе вместо строгого совпадения ролей.
- Рассматривать только «активных» кандидатов (снапшот не старше configurable N дней).

**Результат:** список из 5 `player_id`, значение целевой функции, сохраняется в таблицу `team_proposals` вместе с использованными ограничениями (для воспроизводимости и демонстрации на защите).

### 5.6 Интерфейс (`dashboard/app.py`, Streamlit)

- Таблица игроков с фильтрами по игре, кластеру, диапазону рейтинга.
- Визуализация кластеров (2D-проекция через PCA/t-SNE, точки окрашены по кластеру).
- График изменения рейтинга игрока во времени (по снапшотам).
- Форма запроса оптимального состава: выбор игры, опционально ручная корректировка ограничений (MAX_PER_CLUSTER, MIN_CLUSTERS, размер состава) → кнопка «подобрать состав» → таблица результата с объяснением (значения рейтинга каждого выбранного игрока, роль/кластер).
- Кнопка/индикатор последнего обновления данных (когда планировщик последний раз опрашивал источники).

### 5.7 Backend (`api.py`, FastAPI)

Минимальный REST-слой поверх БД для дашборда и на перспективу — других клиентов:

- `GET /players` — список игроков с фильтрами.
- `GET /players/{id}/history` — история снапшотов/рейтинга.
- `GET /clusters` — текущее разбиение на кластеры.
- `POST /team/optimize` — запуск оптимизатора с параметрами, возврат состава.
- `POST /refresh` — принудительный внеочередной опрос источников (для демонстрации на защите, чтобы не ждать плановый интервал).

## 7. Источники данных

### 7.1 Dota 2 — OpenDota API

- Публичный REST API, часть запросов доступна без ключа.
- Бесплатный уровень: около 60 запросов/минуту и 50 000 запросов/месяц (уточнить на момент реализации — условия тарифов у публичных API могут меняться).
- Ключ (если потребуется больше лимитов) привязывается к способу оплаты на стороне OpenDota.
- Данные: список профессиональных игроков, профиль игрока, история матчей, win/loss, статистика по героям.

### 7.2 CS2 — FACEIT Data API

- REST/JSON, базовый URL `https://open.faceit.com/data/v4`.
- Требует API-ключ (server-side) из FACEIT Developer Portal → App Studio.
- Подтверждённые на этапе подготовки ТЗ эндпоинты перечислены в разделе 5.1.
- Официального публичного API статистики CS2 от Valve нет — FACEIT выбран как основной легальный открытый источник для этой игры.

## 8. Нефункциональные требования

- Система работает полностью локально, не требует постоянного облачного бэкенда.
- Отказ одного источника данных не должен останавливать работу с другим (изоляция ошибок на уровне адаптера).
- API-ключи хранятся только в переменных окружения (`.env`, не в репозитории).
- Логирование обращений к внешним API и ошибок (успех/неуспех, код ответа, оставшийся лимит, если источник его возвращает).
- Добавление новой игры в будущем — это новый адаптер + маппинг в `normalize.py`, без изменений в модулях рейтинга/кластеризации/оптимизации/интерфейса.
- Код обращений к внешним источникам уважает их лимиты запросов (rate limiting/backoff в адаптере, не на уровне планировщика).

## 9. Этапы разработки

1. Инициализация репозитория: структура, `requirements.txt`, `.env.example`, база `CLAUDE.md`/`TECHNICAL_SPEC.md`.

8. FastAPI-эндпоинты поверх БД.

10. Тесты (адаптеры на моках API, нормализация, оптимизатор на синтетических данных), обработка граничных случаев (пустой пул, недостаточно данных для кластеризации, инфизибл-ограничения в оптимизаторе).
11. Отбор и оформление фрагментов кода для приложения курсовой работы (полный код — в приложение, в основном тексте — только показательные фрагменты до 10–12 строк, согласно методичке по оформлению).

## 10. Критерии приёмки

- Пул игроков обеих игр формируется автоматически при первом запуске, без ручного ввода ID.
- В БД накапливается история снапшотов (минимум 3 последовательных опроса с разными timestamp для демонстрации динамики).
- Кластеризация выдаёт устойчивое (не меняющееся хаотично между соседними перерасчётами при отсутствии новых данных) разбиение для обеих игр.
- Рейтинг воспроизводим: одинаковые входные данные дают одинаковый результат, формула объяснима в 2–3 предложениях.
- Оптимизатор возвращает валидный состав из 5 игроков, удовлетворяющий всем заданным ограничениям, либо явно сообщает о невозможности его собрать (infeasible) вместо падения с ошибкой.
- Дашборд открывается локально, показывает игроков, кластеры и позволяет запросить состав без обращения к консоли/коду.
- Отключение/недоступность одного источника не приводит к падению всей системы.

## 11. Ограничения и риски

- Лимиты API (особенно бесплатный ключ FACEIT и лимит OpenDota) ограничивают частоту и объём опроса — планировщик должен закладывать интервалы с запасом.
- Официального API статистики CS2 от Valve не существует — используется сторонний источник (FACEIT), это отраслевое ограничение, а не недоработка проекта.
- Качество кластеризации зависит от объёма накопленной истории: на старте (мало снапшотов) кластеры могут быть нестабильны — рекомендуется зафиксировать в работе период первичного накопления данных перед демонстрацией кластеризации.
- Набор доступных метрик у Dota 2 и CS2 принципиально разный — унифицировать их в одну модель напрямую нельзя, кластеризация и рейтинг считаются раздельно по каждой игре.
- Конкретные `leaderboard_id`/эндпоинты для автосписка топ-игроков FACEIT не зафиксированы в этом ТЗ как факт и должны быть уточнены по актуальной документации на этапе реализации (см. 5.1).

## 12. Возможные расширения (вне текущей версии ТЗ)

- ML-модель прогнозирования вероятности победы состава вместо/в дополнение к простой аддитивной модели рейтинга.
- Live-телеметрия через Game State Integration (CS2/Dota 2) как отдельный источник для матчей, которые видит клиент разработчика — принципиально другая архитектура, не входит в текущий объём.
- Добавление третьей игры (например, LoL/Valorant через Riot API) по той же адаптерной схеме.
