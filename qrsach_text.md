## 1. Архитектура и модульность

Система реализована как последовательность независимых модулей: адаптеры
источников → нормализация → хранилище (SQLite) → рейтинг/кластеризация →
оптимизатор состава → API/дашборд (полная схема — в техническом задании,
раздел 4.1). Ключевое архитектурное требование — модульность: адаптер
источника не знает о существовании кластеризации, а оптимизатор не знает,
откуда взялся рейтинг. Это обеспечивается общим интерфейсом адаптера и тем,
что все вычислительные модули принимают на вход только объект сессии БД и
имя игры, не завязываясь друг на друга напрямую.

**Листинг 1 — общий интерфейс адаптера источника данных**

```python
class BaseCollector(ABC):
    game: str
    min_interval_seconds: float = 0.0

    @abstractmethod
    def fetch_player_pool(self) -> list[dict[str, Any]]:
        """Список кандидатов источника (сырые записи, без нормализации)."""

    @abstractmethod
    def fetch_player_stats(self, external_id: str) -> dict[str, Any]:
        """Сырые данные по одному игроку (без нормализации)."""
```

Оба конкретных адаптера (`OpenDotaCollector` для Dota 2, `FaceitCollector`
для CS2) реализуют этот интерфейс независимо друг от друга. Добавление
третьей игры в будущем потребует только нового класса-адаптера и функции
нормализации, без изменений в остальных модулях — это прямо проверено на
практике при проектировании `scheduler.ingest()` (раздел 4 данного файла).

Отдельно стоит отметить расхождение с черновым вариантом технического
задания: для формирования пула игроков CS2 вместо предполагавшегося
`leaderboard_id`/`hub_id` использован эндпоинт
`/rankings/games/{game}/regions/{region}` — актуальная on-line
Swagger-спецификация FACEIT Data API на момент реализации не подтвердила
черновой вариант, а этот эндпоинт оказался более подходящим для получения
списка топ-игроков без предварительного знания конкретного `hub_id`.

## 2. Соблюдение лимитов внешних API

Оба источника имеют ограничения на частоту запросов (см. ТЗ, раздел 11).
Внутри общего интерфейса адаптера реализованы троттлинг между запросами и
экспоненциальный backoff при ответах 429/5xx.

**Листинг 2 — обработка HTTP 429/5xx с повторными попытками**

```python
if response.status_code == 429 or response.status_code >= 500:
    attempt += 1
    if attempt > max_retries:
        raise SourceUnavailableError(
            f"{self.game}: {url} вернул {response.status_code} после {max_retries} попыток"
        )
    retry_after = response.headers.get("Retry-After")
    forced_delay = float(retry_after) if retry_after else None
    self._sleep_backoff(attempt, backoff_base, forced_delay)
    continue
```

Задержка учитывает заголовок `Retry-After`, если источник его возвращает, и
иначе рассчитывается экспоненциально. Исчерпание всех попыток приводит не к
необработанному исключению, а к контролируемой ошибке `SourceUnavailableError`,
которую вызывающий код (планировщик) перехватывает, изолируя отказ одного
источника от работы другого — нефункциональное требование ТЗ, раздел 8.

## 3. Нормализация: определение роли в Dota 2

OpenDota не отдаёт напрямую позицию игрока (1–5), только лейн конкретного
матча. Роль определяется эвристически: большинством голосов по последним
матчам с учётом лейна и среднего GPM для разделения core/support внутри
safe- и off-лейна (см. ТЗ, раздел 5.2).

**Листинг 3 — эвристика определения роли по данным матчей**

```python
if dominant_lane == LANE_MID:
    return 2

lane_matches = [m for m in votes if m["lane_role"] == dominant_lane]
avg_gpm = sum(m.get("gold_per_min") or 0 for m in lane_matches) / len(lane_matches)
is_core = avg_gpm >= CORE_GPM_THRESHOLD

if dominant_lane == LANE_SAFE:
    return 1 if is_core else 5
return 3 if is_core else 4
```

На практике у значительной доли профессиональных игроков роль остаётся
неопределённой (`None`) — недостаточно матчей с непустым полем `lane_role`
в выборке последних 30 матчей. Это ожидаемое и допустимое по ТЗ поведение
(раздел 5.2 явно разрешает `null`), но оно напрямую влияет на работу
оптимизатора состава: игроки без роли исключаются из пула кандидатов на
конкретный запуск (см. раздел 6).

## 4. Хранение истории: снапшоты, а не перезапись

Каждый успешный опрос источника добавляет новую строку статистики, а не
обновляет предыдущую — это необходимо для последующего анализа динамики
силы игрока и для кластеризации по накопленной истории (ТЗ, раздел 5.1,
критерий приёмки в разделе 10).

**Листинг 4 — модель снапшота статистики игрока**

```python
class PlayerSnapshot(Base):
    __tablename__ = "player_snapshots"

    id: Mapped[int] = mapped_column(Integer, primary_key=True)
    player_id: Mapped[int] = mapped_column(ForeignKey("players.id"), nullable=False)
    taken_at: Mapped[dt.datetime] = mapped_column(DateTime(timezone=True), default=_utcnow, nullable=False)
    metrics: Mapped[dict] = mapped_column(JSON, nullable=False)

    player: Mapped["Player"] = relationship(back_populates="snapshots")
```

Поле `metrics` хранится как JSON, а не как отдельные колонки — это позволяет
Dota 2 и CS2 иметь принципиально разный набор метрик без ветвления схемы
БД, при этом оставаясь в рамках одной таблицы и одного механизма истории.

## 5. Планировщик: периодический опрос источников

APScheduler используется как классический polling-планировщик (не
live-телеметрия), с двумя независимыми job'ами на каждый источник — отказ
одного не блокирует другой.

**Листинг 5 — регистрация периодического job'а опроса**

```python
scheduler.add_job(
    _job_ingest_dota2,
    trigger=IntervalTrigger(minutes=DOTA2_POLL_INTERVAL_MINUTES),
    id="ingest_dota2",
    next_run_time=now,
    max_instances=1,
    coalesce=True,
)
```

Параметр `next_run_time=now` обеспечивает немедленный первый запуск при
старте приложения, не дожидаясь первого интервала — это важно для
демонстрации системы без длительного ожидания.

## 6. Рейтинг игрока: взвешенная сумма z-score

Модель рейтинга сознательно простая и объяснимая (не ML): для каждого
показателя вычисляется z-score относительно других игроков той же игры (и
роли — для Dota 2), затем берётся взвешенная сумма с равными весами по
умолчанию (ТЗ, раздел 5.4).

**Листинг 6 — стандартизация показателя и взвешенное суммирование**

```python
def _zscore(series: pd.Series) -> pd.Series:
    std = series.std(ddof=0)
    if not std or pd.isna(std):
        return pd.Series(0.0, index=series.index)
    return (series - series.mean()) / std


rating_values = sum(z[col] * weight for col, weight in weights.items())
```

Формула воспроизводима (одинаковые входные данные дают одинаковый рейтинг)
и объяснима в двух предложениях, что прямо требуется критериями приёмки ТЗ
(раздел 10): рейтинг — это то, насколько показатели игрока отклоняются от
среднего по сопоставимой группе, суммированное с равными весами.

## 7. Кластеризация: выбор числа кластеров по силуэту

Для сегментации игроков используется KMeans, число кластеров подбирается
автоматически в диапазоне 2–8 по максимуму силуэтного коэффициента, а не
фиксируется константой (ТЗ, раздел 5.3).

**Листинг 7 — подбор числа кластеров по силуэтному коэффициенту**

```python
def _choose_k(x_scaled: np.ndarray, k_range: range) -> int:
    best_k, best_score = k_range.start, -1.0
    for k in k_range:
        if k >= len(x_scaled):
            break
        labels = KMeans(n_clusters=k, n_init=10, random_state=42).fit_predict(x_scaled)
        if len(set(labels)) < 2:
            continue
        score = silhouette_score(x_scaled, labels)
        if score > best_score:
            best_k, best_score = k, score
    return best_k
```

Роль игрока (для Dota 2) намеренно не подаётся кластеризации как признак —
это позволяет использовать полученное разбиение как независимую проверку
формальных ролей 1–5, а не как самоисполняющееся предсказание. Для CS2, где
формальных ролей источник не даёт, кластеры используются оптимизатором как
псевдо-роли (раздел 5.3).

## 8. Оптимизация состава: целочисленное программирование

Подбор состава реализован как классическая задача выбора подмножества
(PuLP) — по аналогии с fantasy-спортом: максимизировать суммарный рейтинг
пяти игроков при выполнении ограничений по ролям (Dota 2) или разнообразию
кластеров (CS2).

**Листинг 8 — целевая функция и ограничения для Dota 2**

```python
problem += pulp.lpSum(c.rating * x[c.player_id] for c in candidates)
problem += pulp.lpSum(x.values()) == team_size

if game == "dota2":
    for role in DOTA2_ROLES:
        role_candidates = [c for c in candidates if c.role == role]
        problem += pulp.lpSum(x[c.player_id] for c in role_candidates) == 1
```

Для CS2, где жёстких ролей нет, разнообразие стилей обеспечивается лимитом
`MAX_PER_CLUSTER` игроков из одного кластера и минимальным числом разных
кластеров `MIN_CLUSTERS` в составе — эти ограничения выражены через
дополнительные бинарные переменные-индикаторы ("used"), что является
стандартным приёмом линеаризации условия "кластер использован хотя бы
одним игроком":

**Листинг 9 — ограничение разнообразия кластеров через переменные-индикаторы**

```python
cluster_sum = pulp.lpSum(x[c.player_id] for c in cluster_candidates)
problem += cluster_sum <= max_per_cluster
used = pulp.LpVariable(f"cluster_used_{label}", cat=pulp.LpBinary)
problem += cluster_sum >= used
problem += cluster_sum <= used * max_per_cluster
cluster_used[label] = used
```

Если ограничения невыполнимы (например, недостаточно кандидатов с нужной
ролью или требуемое число кластеров недостижимо), решатель возвращает
статус `Infeasible`, и система явно сообщает об этом пользователю вместо
падения с необработанным исключением — обязательный критерий приёмки ТЗ
(раздел 10).

## 9. Backend: REST-слой поверх БД

FastAPI предоставляет минимальный REST-интерфейс для дашборда и внешних
клиентов (ТЗ, раздел 5.7), в том числе эндпоинт запуска оптимизатора.

**Листинг 10 — эндпоинт подбора состава**

```python
@app.post("/team/optimize", response_model=TeamOptimizeResponse)
def team_optimize(payload: TeamOptimizeRequest, db: Session = Depends(get_db)) -> TeamOptimizeResponse:
    _check_game(payload.game)
    result = optimize_team(
        db,
        payload.game,
        team_size=payload.team_size,
        max_per_cluster=payload.max_per_cluster,
        min_clusters=payload.min_clusters,
        active_days=payload.active_days,
    )
```

Ответ API дополнительно обогащается никнеймом, ролью и кластером каждого
выбранного игрока (не только внутренним `id`), чтобы результат был сразу
пригоден для отображения на дашборде без дополнительных запросов.

## 11. Тестирование

Тестовый набор (50 тестов, `pytest`) покрывает граничные случаи, явно
указанные в критериях приёмки ТЗ (раздел 10): пустой пул игроков,
недостаточно данных для кластеризации и инфизибл-ограничения в
оптимизаторе. Ниже — регрессионный тест, проверяющий, что кандидат с
устаревшим (старше `active_days`) снапшотом исключается из подбора состава
даже при более высоком рейтинге, чем у активного кандидата:

**Листинг 12 — тест фильтрации неактивных кандидатов**

```python
def test_stale_snapshot_excluded_from_candidates(self, db_session):
    fresh = _add_dota2_player(db_session, external_id="fresh", nickname="Fresh", role="1", rating=1.0)
    _add_dota2_player(
        db_session, external_id="stale", nickname="Stale", role="1", rating=10.0, snapshot_age_days=40
    )
    for role, rating in [("2", 4.0), ("3", 3.0), ("4", 2.0), ("5", 1.0)]:
        _add_dota2_player(db_session, external_id=f"ext-{role}", nickname=f"p{role}", role=role, rating=rating)

    result = optimize_team(db_session, "dota2", active_days=30)
    assert result.feasible
    assert fresh.id in result.player_ids
```

Для HTTP-адаптеров сеть не используется вовсе — запросы подменяются через
`httpx.MockTransport`, что позволяет детерминированно проверить в том числе
поведение при HTTP 429 и повторных попытках без реального ожидания
(задержки backoff подменяются на no-op на время теста).

## 12. Результаты практической проверки

Помимо модульных тестов, каждый модуль был проверен вживую на реальных
данных источников на соответствующем этапе разработки:

- пул профессиональных игроков Dota 2 и пул топ-игроков CS2 по региону
  формируются автоматически при первом запуске, без ручного ввода
  идентификаторов (критерий приёмки ТЗ, раздел 10);
- при накоплении нескольких снапшотов рейтинг и кластеризация
  пересчитываются воспроизводимо;
- оптимизатор на реальной выборке (30 игроков Dota 2, 40 игроков CS2)
  вернул для CS2 валидный состав из пяти игроков, представляющих три разных
  кластера, с суммарным рейтингом около 20,1; для Dota 2 в этой же выборке
  честно вернул `infeasible`, так как ни у одного из отобранных игроков не
  определилась роль "2" (мид) — что является прямым следствием эвристики из
  раздела 3 данного файла, а не ошибкой оптимизатора;
- полный тестовый набор (50 тестов) проходит без ошибок, что вместе с
  живыми проверками закрывает критерии приёмки ТЗ, раздел 10. Единственный
  оставшийся пункт плана разработки (ТЗ, раздел 9) — отбор фрагментов кода
  и оформление текста работы (этап 11), которому и посвящён настоящий файл.
