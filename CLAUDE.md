# CLAUDE.md

Контекст для Claude Code при работе с этим репозиторием. Перед реализацией нового модуля — сверяйся с `TECHNICAL_SPEC.md` (полное техническое задание), не придумывай требования от себя.

## О проекте

Практическая часть курсовой работы «Система подбора киберспортсменов, их сегментация, кластеризация, система выстраивания оптимальной команды для побед на различных чемпионатах».

Система собирает статистику игроков Dota 2 (OpenDota API) и CS2 (FACEIT Data API) из открытых источников, приводит её к единой схеме, считает внутренний рейтинг силы игрока, кластеризует игроков по стилю/уровню игры и подбирает оптимальный состав команды (5 человек) под ограничения по ролям/кластерам через целочисленное программирование. Работает локально, интерфейс — Streamlit.

## Стек

- Python 3.11+
- FastAPI — backend/API поверх собранных данных
- APScheduler — периодический опрос источников (это не live-телеметрия, а polling по расписанию)
- SQLAlchemy + SQLite — хранение
- httpx — HTTP-клиент для внешних API
- pandas, scikit-learn — сегментация/кластеризация (KMeans/DBSCAN, PCA), расчёт признаков
- PuLP — оптимизация состава (целочисленное программирование)
- Streamlit — интерфейс/дашборд
- pytest — тесты

## Структура репозитория

Предлагаемая, можно адаптировать по ходу разработки — главное держать источники данных изолированными друг от друга и от остальной логики.

```
project/
├── CLAUDE.md
├── TECHNICAL_SPEC.md
├── .env.example
├── requirements.txt
├── app/
│   ├── collectors/            # адаптеры источников, один файл — один источник
│   │   ├── base.py            # общий интерфейс адаптера
│   │   ├── opendota.py        # Dota 2
│   │   └── faceit.py          # CS2
│   ├── scheduler.py           # APScheduler: job'ы опроса источников
│   ├── normalize.py           # сырые ответы API -> единая схема
│   ├── models.py               # SQLAlchemy-модели (players, snapshots, clusters, ratings, team_proposals)
│   ├── db.py                    # engine/session
│   ├── clustering.py           # сегментация/кластеризация
│   ├── rating.py                # расчёт внутреннего рейтинга игрока
│   ├── optimizer.py            # подбор оптимального состава (PuLP)
│   └── api.py                   # FastAPI роуты
├── dashboard/
│   └── app.py                   # Streamlit-интерфейс
├── data/
│   └── app.db                    # SQLite, в .gitignore
└── tests/
    ├── test_collectors.py
    ├── test_normalize.py
    └── test_optimizer.py
```

## Команды

```bash
pip install -r requirements.txt        # установка зависимостей
uvicorn app.api:app --reload           # backend + запуск планировщика
streamlit run dashboard/app.py         # дашборд
pytest                                  # тесты
```

## Переменные окружения (`.env`, не коммитить)

- `FACEIT_API_KEY` — server-side ключ из FACEIT Developer Portal (App Studio)
- `OPENDOTA_API_KEY` — опционально, для базового использования не обязателен (публичный лимит ~60 запросов/мин, 50 000/мес)

## Правила разработки

- Каждый источник данных — отдельный адаптер, реализующий общий интерфейс из `collectors/base.py` (например `fetch_player_pool()`, `fetch_player_stats(external_id)`). Добавление новой игры в будущем не должно требовать изменений вне `collectors/` и `normalize.py`.
- Сырые ответы внешних API никогда не уходят наружу напрямую — только через `normalize.py` в единую схему, описанную в `TECHNICAL_SPEC.md`.
- Уважать лимиты запросов источников: задержки/бэкофф внутри адаптеров, не увеличивать частоту опроса без явной необходимости.
- Не парсить HLTV и подобные сайты, где это запрещено правилами использования — только официальные/открытые API (OpenDota, FACEIT).
- Перед тем как жёстко зашивать конкретные `leaderboard_id`/эндпоинты FACEIT для пула игроков — свериться с актуальной документацией (docs.faceit.com), в ТЗ они помечены как «уточнить при реализации».
- Каждый снапшот статистики пишется как новая строка с timestamp, а не перезаписывает предыдущую — история важна для кластеризации и рейтинга.
- Код и комментарии — русский или английский, но единообразно в пределах одного файла.
- Для новых модулей — сначала минимальный рабочий вертикальный срез (один источник, без оптимизаций), затем расширение, а не наоборот.
