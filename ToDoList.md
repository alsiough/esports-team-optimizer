     `lane_role` (большинство голосов среди последних матчей, нужно ≥3

     допускается ТЗ 5.2.
   - CS2: `winrate, kd, adr, hs_pct` из `lifetime`-объекта FACEIT. ВАЖНО:
     `lifetime["K/D Ratio"]` и `lifetime["Total Headshots %"]` — битые
     агрегаты (наблюдались абсурдные значения вроде 10331.88) — используются
     `"Average K/D Ratio"` и `"Average Headshots %"`. Роль для CS2 всегда
     `None` (будет через кластеризацию).
6. **`app/models.py`** / **`app/db.py`** — SQLAlchemy-схема (players,
   player_snapshots, clusters, ratings, team_proposals) точно по ТЗ 6.2,
   SQLite, `.env` подгружается через `python-dotenv` в `db.py`.
7. **`app/scheduler.py`** — `ingest()` — общая функция опроса+нормализации+
   записи (тонкие обёртки `ingest_dota2`/`ingest_cs2`, чтобы третья игра в
   будущем не требовала копипасты — см. ТЗ 4.2/8), плюс `create_scheduler()`
   с APScheduler `BackgroundScheduler`, двумя независимыми interval-job'ами,
   immediate first run, изоляция ошибок на уровне job'а. Каждый job в конце
   вызывает пересчёт рейтинга. Конфигурация через env
   (`DOTA2_POLL_INTERVAL_MINUTES`, `DOTA2_POOL_LIMIT`, `CS2_POLL_INTERVAL_MINUTES`,
   `CS2_POOL_LIMIT`, `FACEIT_REGION`) — НЕ документированы в CLAUDE.md как
   обязательные, это опциональные тюнинг-параметры с дефолтами.
8. **`app/rating.py`** — `compute_ratings(session, game)`: z-score метрик
   последнего снапшота каждого игрока относительно других игроков той же
   игры (для Dota 2 — в пределах роли; для CS2 — вся игра, кластеров пока
   нет), веса равные (константы `DOTA2_WEIGHTS`/`CS2_WEIGHTS`, легко
   поменять). История не перезаписывается. Воспроизводимость проверена
   вживую.
9. **`app/clustering.py`** — `compute_clusters(session, game)`: агрегация
   метрик по снапшотам за период (среднее, по умолчанию 30 дней),
   стандартизация, KMeans с числом кластеров по силуэту (диапазон 2-8, роль
   НЕ подаётся как признак — кластеры остаются независимой проверкой ролей).
   `detect_outliers_dbscan()` и `pca_2d()` — вспомогательные, в БД не пишут.
   История кластеров копится, не перезаписывается.
10. **`app/optimizer.py`** — `optimize_team(session, game, ...)`: подбор
    состава из 5 игроков через PuLP (ЦП), см. ТЗ 5.5. Разбит на
    `optimize_candidates()` (чистая функция над списком `Candidate`, без БД —

    `MIN_CLUSTERS`/`MAX_PER_CLUSTER`, отсутствие роли) и `optimize_team()`
    (тянет кандидатов из БД — последний рейтинг из `ratings`, активность по

    результат в `team_proposals`). Dota 2: ровно один игрок на роль 1-5,
    кандидаты без роли исключены. CS2: `MAX_PER_CLUSTER` (default 2) /
    `MIN_CLUSTERS` (default 3) через bigM-индикаторы бинарных переменных на
    кластер. Инфизибл — `OptimizationResult(feasible=False, message=...)`,
    без исключений и без записи в БД.

## Как тестировать вживую (паттерн, использованный на этапах 2-6)

1. `os.environ["DATABASE_URL"] = "sqlite:///<временный путь>"` ДО импорта
   `app.db` — чтобы не трогать `data/app.db` в самом репозитории.
2. `from dotenv import load_dotenv; load_dotenv(r"F:\pets\diank-qrsach\.env")`
   — если нужен `FACEIT_API_KEY`.
3. `PYTHONIOENCODING=utf-8` при запуске через Bash-обёртку на Windows —
   иначе падает на никнеймах с не-cp1251 символами (напр. китайские ники)
   при печати в консоль.
4. Пул для смоука брать маленький (`pool_limit=5..20`) — не гонять весь
   пул OpenDota (тысячи про-игроков, бюджет 50k запросов/мес).

## 2026-07-13: расширение сбора данных (больше игроков/регионов/истории)

По просьбе пользователя "как собирать больше данных" — три независимых улучшения:

- **Больше регионов CS2** — `FaceitCollector` (`app/collectors/faceit.py`)
  теперь принимает `regions: str | list[str]` вместо одного `region: str`.
  Валидные коды регионов проверены вживую прямым запросом к
  `GET /rankings/games/cs2/regions/{region}` с реальным ключом (не по
  документации — она не перечисляет допустимые значения): непустой список
  вернули `EU`, `NA`, `SA`, `OCE`, `SEA` (сохранены в
  `faceit.py:KNOWN_REGIONS`); правдоподобные варианты `US`, `AF`, `AS`, `OC`,
  `ASIA`, `ME`, `MENA`, `APAC` дают `200 OK`, но с пустым `items` — эндпоинт
  не возвращает ошибку на неизвестный код региона, тихо отдаёт пустой список,
  так что доверять надо только реально проверенным значениям.
  `fetch_player_pool()` объединяет региональные топы **вперемешку
  (round-robin)**, а не один за другим — иначе обрезка пула до
  `pool_limit` в `scheduler.ingest()` забрала бы только первый регион.
  Дедуп по `player_id` на случай пересечений между регионами. Конфигурация —
  новая env-переменная `FACEIT_REGIONS` (список через запятую, дефолт — все
  5 проверенных регионов); старая `FACEIT_REGION` (единственное число)
  заменена, использовалась только как опциональный тюнинг-параметр, нигде
  не документирована как обязательная.
- **Больше игроков в пуле** — `DOTA2_POOL_LIMIT` 20→30,
  `CS2_POOL_LIMIT` 50→100 (дефолты в `app/scheduler.py`). Для OpenDota
  сознательно не подняли сильнее — 3 запроса/игрока (`wl`/`totals`/`matches`)
  при непрерывной работе 24/7 и так подводят к месячному бюджету 50k, а это
  локальный демо-проект, не постоянно работающий сервис (см. комментарий в
  коде). Для FACEIT лимит был занижен без явной причины (публичного лимита
  API не документировано) — подняли смелее.
- **Больше истории снапшотов во времени** — это НЕ конфигурационный вопрос:
  каждый вызов `refresh_game()`/planned job добавляет ровно один новый
  снапшот на игрока (см. раздел 4 данного файла, история не перезаписывается
  по дизайну), но реальная динамика показателей появляется только когда
  между опросами проходит настоящее время (сыграны новые матчи) — опрос
  дважды подряд за минуты даст два снапшота с почти одинаковыми метриками.
  Чтобы накопить содержательную историю к защите: либо держать
  `uvicorn app.api:app` запущенным подольше (планировщик сам опрашивает
  каждые `DOTA2_POLL_INTERVAL_MINUTES`/`CS2_POLL_INTERVAL_MINUTES`, дефолт
  60 минут, с немедленным первым запуском), либо периодически жать
  "Опросить источник сейчас" в дашборде / дёргать `POST /refresh` в течение
  нескольких дней перед защитой.
  Проверено живым прогоном: `refresh_game(session, "cs2")` с новыми
  настройками (5 регионов, pool_limit=100) добавил 100 новых снапшотов,
  итого 130 уникальных cs2-игроков в БД (было 50); `refresh_game(session,
  "dota2")` с `pool_limit=30` — без изменений (тот же пул, что и раньше).
  Оптимизатор остался `feasible` для обеих игр, целевая функция CS2 выросла
  (19.6 → 29.0) за счёт более разнообразного и большого пула кандидатов.
  Добавлены 2 теста на `FaceitCollector.fetch_player_pool()` (round-robin
  вперемешку между регионами, дедуп по `player_id` при пересечении) —
  полный набор (52 теста) проходит.

## 2026-07-13: диагностика и фикс `Infeasible` в оптимизаторе

Пользователь столкнулся с `Infeasible` на реальных данных для обеих игр.
Диагностика на живой `data/app.db` (без моков) показала две разные причины:

- **Dota 2**: из 20 игроков в пуле ни у одного не определилась роль "2"
  (мид) — см. уже описанную выше проблему с пустым `lane_role` у OpenDota.
  Фикс: `MATCHES_LIMIT` в `app/collectors/opendota.py` поднят с 30 до 100
  (бесплатно — это один и тот же HTTP-запрос `/matches`, просто с большим
  `limit`, доп. нагрузки на бюджет OpenDota нет). Проверено живым переопросом
  30 игроков: роль "2" появилась (1 игрок), `optimize_team(db, "dota2")`
  стал `feasible` (objective ≈13.9). `DOTA2_POOL_LIMIT`/интервалы намеренно
  НЕ трогали — это уже отдельная статья бюджета (см. пункт выше).
- **CS2**: кластеризация на 50 игроках в этом прогоне нашла всего 2
  кластера (KMeans сам выбирает k по силуэту, диапазон 2-8), а дефолты
  оптимизатора (`DEFAULT_MIN_CLUSTERS=3`, `DEFAULT_MAX_PER_CLUSTER=2`) в
  паре с 2 кластерами математически не позволяют набрать 5 игроков (2×2=4
  максимум) — не баг оптимизатора, а нежизнеспособная комбинация дефолтов
  при малом числе кластеров. По решению пользователя дефолты в
  `app/optimizer.py` изменены: `DEFAULT_MIN_CLUSTERS` 3 → 2,
  `DEFAULT_MAX_PER_CLUSTER` 2 → 3 (это сознательное отступление от
  черновых цифр ТЗ 5.5 "по умолчанию 2"/"по умолчанию 3" — параметры
  остаются настраиваемыми через форму дашборда/API на конкретный запрос).
  Проверено: `optimize_team(db, "cs2")` стал `feasible` (objective ≈19.6).
  Оба параметра по-прежнему тюнинговые, не финальные — пересмотреть при
  накоплении более богатой выборки кластеров.
- Два теста в `tests/test_optimizer.py`
  (`TestOptimizeCandidatesCs2.test_respects_max_per_cluster_and_min_clusters`,
  `test_min_clusters_unreachable_is_infeasible`) неявно полагались на старые
  значения дефолтов — переписаны так, чтобы явно передавать
  `max_per_cluster`/`min_clusters` в вызов, а не полагаться на константы
  модуля. Полный набор (50 тестов) проходит.
