## Сделано (этапы 1-9, все запушены)

1. **Структура репозитория** — `app/`, `dashboard/`, `data/`, `tests/`,
   requirements.txt, .env.example, .gitignore.
2. **Адаптер OpenDota** (`app/collectors/opendota.py`) — `/proPlayers` (пул),
   `/players/{id}/wl` + `/totals` + `/matches?project=...` (стата). Проверено
   вживую на реальном API (без ключа, публичный лимит).
3. **Адаптер FACEIT** (`app/collectors/faceit.py`) — `/rankings/games/cs2/regions/{region}`
   для пула (НЕ `leaderboard_id`/`hub_id`, как черновым вариантом
   предполагало ТЗ — это осознанное отступление, живая swagger-спека
   `https://open.faceit.com/data/v4/docs/swagger.json` показала более
   подходящий endpoint), `/players/{id}/stats/{game_id}` для статы,
   `Authorization: Bearer <key>`. Проверено вживую с реальным ключом.
4. **`app/collectors/base.py`** — общий интерфейс + throttling + backoff на
   429/5xx, используется обоими адаптерами.
5. **`app/normalize.py`** — сырые ответы → единая схема снапшота:
   - Dota 2: `winrate, kda, gpm, xpm, hero_damage`; роль 1-5 — эвристика по
     `lane_role` (большинство голосов среди последних матчей, нужно ≥3
     матчей с непустым lane_role) + gpm-порог 400 для разделения core/support
     внутри safe/off лейна. Часто остаётся `None` — у большинства
     непарсенных матчей OpenDota `lane_role` пустой, это ожидаемо и
     допускается ТЗ 5.2.
   - CS2: `winrate, kd, adr, hs_pct` из `lifetime`-объекта FACEIT. ВАЖНО:
     `lifetime["K/D Ratio"]` и `lifetime["Total Headshots %"]` — битые
     агрегаты (наблюдались абсурдные значения вроде 10331.88) — используются
     `"Average K/D Ratio"` и `"Average Headshots %"`. Роль для CS2 всегда
     `None` (будет через кластеризацию).
6. **`app/models.py`** / **`app/db.py`** — SQLAlchemy-схема (players,
   player_snapshots, clusters, ratings, team_proposals) точно по ТЗ 6.2,
   SQLite, `.env` подгружается через `python-dotenv` в `db.py`.
7. **`app/scheduler.py`** — `ingest()` — общая функция опроса+нормализации+
   записи (тонкие обёртки `ingest_dota2`/`ingest_cs2`, чтобы третья игра в
   будущем не требовала копипасты — см. ТЗ 4.2/8), плюс `create_scheduler()`
   с APScheduler `BackgroundScheduler`, двумя независимыми interval-job'ами,
   immediate first run, изоляция ошибок на уровне job'а. Каждый job в конце
   вызывает пересчёт рейтинга. Конфигурация через env
   (`DOTA2_POLL_INTERVAL_MINUTES`, `DOTA2_POOL_LIMIT`, `CS2_POLL_INTERVAL_MINUTES`,
   `CS2_POOL_LIMIT`, `FACEIT_REGION`) — НЕ документированы в CLAUDE.md как
   обязательные, это опциональные тюнинг-параметры с дефолтами.
8. **`app/rating.py`** — `compute_ratings(session, game)`: z-score метрик
   последнего снапшота каждого игрока относительно других игроков той же
   игры (для Dota 2 — в пределах роли; для CS2 — вся игра, кластеров пока
   нет), веса равные (константы `DOTA2_WEIGHTS`/`CS2_WEIGHTS`, легко
   поменять). История не перезаписывается. Воспроизводимость проверена
   вживую.
9. **`app/clustering.py`** — `compute_clusters(session, game)`: агрегация
   метрик по снапшотам за период (среднее, по умолчанию 30 дней),
   стандартизация, KMeans с числом кластеров по силуэту (диапазон 2-8, роль
   НЕ подаётся как признак — кластеры остаются независимой проверкой ролей).
   `detect_outliers_dbscan()` и `pca_2d()` — вспомогательные, в БД не пишут.
   История кластеров копится, не перезаписывается.
10. **`app/optimizer.py`** — `optimize_team(session, game, ...)`: подбор
    состава из 5 игроков через PuLP (ЦП), см. ТЗ 5.5. Разбит на
    `optimize_candidates()` (чистая функция над списком `Candidate`, без БД —
    проверена на синтетике: базовый успех, нехватка кандидатов, недостижимые
    `MIN_CLUSTERS`/`MAX_PER_CLUSTER`, отсутствие роли) и `optimize_team()`
    (тянет кандидатов из БД — последний рейтинг из `ratings`, активность по
    последнему снапшоту, для CS2 последний кластер из `clusters`; сохраняет
    результат в `team_proposals`). Dota 2: ровно один игрок на роль 1-5,
    кандидаты без роли исключены. CS2: `MAX_PER_CLUSTER` (default 2) /
    `MIN_CLUSTERS` (default 3) через bigM-индикаторы бинарных переменных на
    кластер. Инфизибл — `OptimizationResult(feasible=False, message=...)`,
    без исключений и без записи в БД.
    Живой прогон на реальных данных (30 dota2 + 40 cs2 через временную
    SQLite): CS2 — валидный состав, 3 разных кластера, сохранён в
    `team_proposals`. Dota 2 — `infeasible`, т.к. в этой выборке ни у одного
    игрока не определилась роль "2" (мид) — соответствует уже известной
    проблеме с частым `None` у роли (см. "Открытые вопросы" ниже), не баг
    оптимизатора.
    Попутно обнаружен и исправлен баг окружения: локальный `.env` (не в git)
    содержал `OPENDOTA_API_KEY=your_opendota_api_key_here(unnecessary)` —
    буквальный текст-плейсхолдер вместо пустого значения, из-за чего OpenDota
    API отвечал 400 на любой запрос. Исправлено на пустое значение (ключ и
    не нужен, см. CLAUDE.md).
11. **`app/api.py`** — FastAPI поверх БД, ровно 5 эндпоинтов из ТЗ 5.7:
    `GET /players` (фильтры `game` обязателен, `cluster`, `min_rating`,
    `max_rating` — рейтинг и кластер подтягиваются через новые

    - Проверено: `TestClient` с замоканным `create_scheduler` (чтобы не

## Как тестировать вживую (паттерн, использованный на этапах 2-6)

1. `os.environ["DATABASE_URL"] = "sqlite:///<временный путь>"` ДО импорта
   `app.db` — чтобы не трогать `data/app.db` в самом репозитории.
2. `from dotenv import load_dotenv; load_dotenv(r"F:\pets\diank-qrsach\.env")`
   — если нужен `FACEIT_API_KEY`.
3. `PYTHONIOENCODING=utf-8` при запуске через Bash-обёртку на Windows —
   иначе падает на никнеймах с не-cp1251 символами (напр. китайские ники)
   при печати в консоль.
4. Пул для смоука брать маленький (`pool_limit=5..20`) — не гонять весь
   пул OpenDota (тысячи про-игроков, бюджет 50k запросов/мес).

## Открытые вопросы / на заметку

- Дефолты `DOTA2_POOL_LIMIT=20` / `CS2_POOL_LIMIT=50` и интервалы 60 минут —
  прикидка с запасом, не финальное тюнинговое решение. Можно пересмотреть

  НЕ реализована — нет однозначного поля в FACEIT `lifetime` под неё, решили
  не натягивать случайное поле под чужое название. Финальный набор метрик
  CS2: winrate, kd, adr, hs_pct.
- Роль Dota 2 у многих про-игроков остаётся `None` (мало матчей с
  непустым `lane_role` в выборке последних 30) — подтверждено и на этапе 7:
  живой прогон на 30 игроках вообще не дал ни одного с ролью "2" (мид), из-за
  чего оптимизатор для dota2 вернул `infeasible`. Стоит пересмотреть на одном
  из следующих этапов: то ли увеличивать `MATCHES_LIMIT`/размер пула в опросе,
  то ли смириться и просто исключать таких игроков из пула для оптимизатора
  (сейчас так и задумано по ТЗ 5.5, `optimizer.py` это уже делает).
