     допускается ТЗ 5.2.
   - CS2: `winrate, kd, adr, hs_pct` из `lifetime`-объекта FACEIT. ВАЖНО:
     `lifetime["K/D Ratio"]` и `lifetime["Total Headshots %"]` — битые
     агрегаты (наблюдались абсурдные значения вроде 10331.88) — используются
     `"Average K/D Ratio"` и `"Average Headshots %"`. Роль для CS2 всегда
     `None` (будет через кластеризацию).
6. **`app/models.py`** / **`app/db.py`** — SQLAlchemy-схема (players,
   player_snapshots, clusters, ratings, team_proposals) точно по ТЗ 6.2,
   SQLite, `.env` подгружается через `python-dotenv` в `db.py`.
7. **`app/scheduler.py`** — `ingest()` — общая функция опроса+нормализации+
   записи (тонкие обёртки `ingest_dota2`/`ingest_cs2`, чтобы третья игра в
   будущем не требовала копипасты — см. ТЗ 4.2/8), плюс `create_scheduler()`
   с APScheduler `BackgroundScheduler`, двумя независимыми interval-job'ами,
   immediate first run, изоляция ошибок на уровне job'а. Каждый job в конце
   вызывает пересчёт рейтинга. Конфигурация через env
   (`DOTA2_POLL_INTERVAL_MINUTES`, `DOTA2_POOL_LIMIT`, `CS2_POLL_INTERVAL_MINUTES`,
   `CS2_POOL_LIMIT`, `FACEIT_REGION`) — НЕ документированы в CLAUDE.md как
   обязательные, это опциональные тюнинг-параметры с дефолтами.
8. **`app/rating.py`** — `compute_ratings(session, game)`: z-score метрик
   последнего снапшота каждого игрока относительно других игроков той же
   игры (для Dota 2 — в пределах роли; для CS2 — вся игра, кластеров пока
   нет), веса равные (константы `DOTA2_WEIGHTS`/`CS2_WEIGHTS`, легко

9. **`app/clustering.py`** — `compute_clusters(session, game)`: агрегация

### Этап 7 — `app/optimizer.py` (PuLP), СЛЕДУЮЩИЙ ШАГ

Целочисленное программирование, см. ТЗ 5.5 дословно:
- Переменные `x_i ∈ {0,1}` на кандидата одной игры (Dota2 и CS2 не
  смешивать).
- Максимизировать `Σ rating_i * x_i` (rating — последнее значение из таблицы
  `ratings`, см. `app/rating.py`).
- Ограничения:
  - `Σ x_i = 5`.
  - Dota 2: ровно один игрок на каждую роль 1-5 (`player.role`, строка
    "1".."5"); кандидаты без роли — исключаются из пула на этот запуск
    (роль иначе не гарантируется уникальной).
  - CS2: не более `MAX_PER_CLUSTER` (default 2) игроков из одного кластера
    и не менее `MIN_CLUSTERS` (default 3) разных кластеров — кластер брать
    из последней записи в таблице `clusters` для игрока и алгоритма (нужно
    решить, как выбирать "последний" алгоритм/расчёт, если их несколько —
    скорее всего по `computed_at` в пределах одного `algorithm`).
  - Только "активные" кандидаты — снапшот не старше configurable N дней.
- Результат: список из 5 `player_id`, `objective_value`, сохранить в
  `team_proposals` (game, created_at, constraints JSON, player_ids JSON,
  objective_value).
- Инфизибл-ограничения — вернуть явный сигнал/сообщение, НЕ падать с
  необработанным исключением (ТЗ 10, критерий приёмки).
- Тестировать: сначала на синтетических данных (легко проверить руками),

## Как тестировать вживую (паттерн, использованный на этапах 2-6)

1. `os.environ["DATABASE_URL"] = "sqlite:///<временный путь>"` ДО импорта
   `app.db` — чтобы не трогать `data/app.db` в самом репозитории.
2. `from dotenv import load_dotenv; load_dotenv(r"F:\pets\diank-qrsach\.env")`
   — если нужен `FACEIT_API_KEY`.
3. `PYTHONIOENCODING=utf-8` при запуске через Bash-обёртку на Windows —
   иначе падает на никнеймах с не-cp1251 символами (напр. китайские ники)
   при печати в консоль.
4. Пул для смоука брать маленький (`pool_limit=5..20`) — не гонять весь
   пул OpenDota (тысячи про-игроков, бюджет 50k запросов/мес).

  не натягивать случайное поле под чужое название. Финальный набор метрик
  CS2: winrate, kd, adr, hs_pct.

  непустым `lane_role` в выборке последних 30). Возможно, стоит на этапе 7

  задумано по ТЗ 5.5).
